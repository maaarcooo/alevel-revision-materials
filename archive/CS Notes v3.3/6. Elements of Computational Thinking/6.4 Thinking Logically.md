# 6.4 Thinking Logically

## What is Thinking Logically?

**Thinking logically** involves identifying where decisions need to be made within a solution, determining the conditions that affect those decisions, and understanding how decisions alter the flow of a program. It enables planning for different scenarios by providing foresight of all decision points throughout a solution.

A **decision** is a result reached after some consideration. Good decision making is fundamental to effective problem solving.

## Structured Programming Techniques

Most programming languages use **structured programming techniques** to control the flow of execution:

**Sequence**: Statements executed one after another in order.

**Selection**: Decision making that directs program flow based on conditions. Implemented using `if/then/else` or `switch/case` statements.

**Iteration**: Repeating a sequence of instructions based on a stopping condition. Implemented using loops (`for`, `while`, `do while`, `do until`).

These techniques aid the **readability**, **understanding**, and **maintainability** of code.

**Block-structured languages** (e.g., Python) use only these three constructs to control execution flow. Each block of code should ideally have a **single entry and exit point** to prevent unintended consequences such as entering or leaving subroutines early or unintentional branching.

## Identifying Decision Points

When designing solutions, decisions must be made at multiple levels:

**Design-level decisions**:
- Which programming paradigm to use
- Which programming language to use
- What input/output devices are needed
- How to interact with hardware/software components

**Runtime decisions**:
- Conditions evaluated during program execution
- User choices that determine program behaviour

### Methods for Planning Decisions

**Flowcharts**: Visually show flow of control. Decisions are represented by **diamond-shaped symbols** with two outputs (Yes/No or True/False). Clear and easy to follow but time-consuming to create.

**Pseudocode**: Mimics programming language constructs without strict syntax. More accurately represents code structure. Any expression is acceptable provided the meaning is clear.

**Structured English**: Alternative to pseudocode but typically more verbose and less precise.

Planning algorithms before coding helps identify decision points and allows solutions to be implemented in any language, avoiding language-specific syntax issues.

## Conditions Affecting Decision Outcomes

When making decisions, evaluate these key factors and prioritise them:

**Effectiveness**: What achieves the best results for the problem?

**Convenience**: What is most practical to implement?

**Feasibility**: Is this option reasonable given constraints (time, resources, skill level)?

The priority of these factors varies depending on the **purpose** and **end-users** of the software. Ordering conditions from most to least important helps identify the best approach.

## How Decisions Affect Program Flow

Decisions direct the **flow of control** through different execution paths. Different choices produce completely different routes through a program.

### Selection Structures

**Selection** (also called **branching**) directs program flow dependent on a Boolean condition or set of conditions.

**If/then/else statements**: Evaluate conditions sequentially until one is true.

```
if condition1 then
    // execute if condition1 is true
elseif condition2 then
    // execute if condition2 is true
else
    // execute if no conditions are true
endif
```

**Switch/case statements**: Match a value against multiple cases.

```
switch variable:
    case value1:
        // execute for value1
    case value2:
        // execute for value2
    default:
        // execute if no case matches
endswitch
```

Both structures are functionally equivalent but switch/case can be cleaner when comparing a single variable against multiple values.

### Iteration Structures

**Iteration** repeats instructions based on a stopping **Boolean condition**.

**While loops**: Condition checked before each iteration. May execute zero times.

**Repeat/until loops**: Condition checked after each iteration. Always executes at least once.

**For loops**: Iterate a fixed number of times or over a collection.

### Nested Structures

Iteration and selection statements can be **nested** within each other, creating complex decision structures. Each nested if statement or loop affects the flow of the program and creates additional decision points.

## Boolean Conditions

Most program errors occur when evaluating **Boolean conditions**. Care is required when creating conditions, especially complex ones with multiple clauses.

### Comparison Operators

| Operator | Meaning |
|----------|---------|
| `==` | Equal to |
| `!=` | Not equal to |
| `>` | Greater than |
| `>=` | Greater than or equal to |
| `<` | Less than |
| `<=` | Less than or equal to |

### Logical Operators

| Operator | Meaning |
|----------|---------|
| `AND` | Both conditions must be true |
| `OR` | Either or both conditions must be true |
| `NOT` | Inverts the condition (true becomes false) |
| `XOR` | Exactly one condition must be true (not both) |

### Example: Compound Condition

A teenager wants to see a 15-rated film costing £8. They need to be at least 15 years old AND have enough money:

```
if age >= 15 AND money >= 8 then
    entry = True
endif
```

Both conditions must evaluate to true for entry to be granted.

## Using Break Statements

The **break** instruction exits a loop early when a condition is met. While powerful for controlling flow, break statements can cause unintended problems if misused. Use with caution and ensure the logic is clear.

## Key Points Summary

- **Thinking logically** means identifying decision points, understanding conditions, and tracing how decisions affect program flow
- **Structured programming** uses sequence, selection, and iteration to control execution
- **Flowcharts** visually represent decisions; **pseudocode** represents logic without syntax constraints
- Evaluate decisions based on **effectiveness**, **convenience**, and **feasibility**
- **Boolean conditions** determine which path a program takes at decision points
- Most errors occur in Boolean condition evaluation — take care with complex conditions
- **Selection** directs flow based on conditions; **iteration** repeats based on stopping conditions
- Nested structures create multiple decision points affecting program flow
